📋 YAML Syntax Reference

Top-level Structure

title: "My Form Title"
description: "Optional description"

github:                          # Optional — GitHub Actions dispatch
  repository: "org/repo"
  workflow:   "workflow-file.yml"
  event_type: "my_event_type"

ansible:                         # Optional — Ansible Tower / AWX
  tower_url:       "https://tower.example.com"
  job_template_id: 42

fields:
  - name: "fieldName"
    label: "Field Label"
    type:  "text"
    ...

Ansible Tower — ansible

Triggers an Ansible Tower / AWX job template. Form data is passed as extra_vars. Enter your Tower API token in the left panel (not saved).

PropertyRequiredDescription
tower_urlYesBase URL of your Tower/AWX instance (e.g. https://tower.example.com)
job_template_idYesNumeric ID of the job template to launch

Both github: and ansible: can coexist in the same form — each gets its own payload section and dispatch button.

Field Types

typeDescription
textSingle-line text input
emailEmail input (browser-validated)
numberNumeric input — supports min / max
datetime-localDate and time picker
textareaMulti-line text input
dropdownSingle-select — requires options
checkboxMulti-select — requires options; payload value is an array

Field Properties

PropertyRequiredDescription
nameYesUnique key used in the payload
labelYesLabel shown above the field
typeYesOne of the types listed above
requiredNoMarks field as mandatory. Default: false
placeholderNoHint text inside the input
defaultNoPre-filled value on load
optionsdropdown / checkboxList of { value, label } choices
min / maxNoNumeric bounds (number type only)
noteNo💡 Helper text shown below the field
show_ifNoConditional visibility rule — see below

Conditional Logic — show_if

Hidden fields are excluded from the payload and their values are cleared when hidden.

show_if:
  field:    "sourceFieldName"  # field to watch
  operator: "equals"           # optional, defaults to "equals"
  value:    "targetValue"      # value to match
OperatorDescription
equals (default)Value exactly matches value
not_equalsValue does not match value
containsCheckbox includes value, or text contains substring
not_emptyField has any non-empty value (value not needed)

Chaining: A conditional field can itself be the source of another show_if, enabling multi-level branching.

Options Syntax (static)

options:
  - value: "payload_value"   # written to the GitHub Actions payload
    label: "Displayed label" # shown to the user in the UI

Form-level Environment Variables — env

Declare key/value pairs at the top of your YAML. They are saved with the form and can be referenced anywhere with {{env:VAR_NAME}} in source.url or source.headers.

env:
  DT_API_KEY: "dt0c01.xxxx"
  DT_ENV_ID:  "abc123.live.dynatrace.com"

Resolution order: form-level env values override server environment variables with the same name. Values are stored in plaintext alongside the form — use server environment variables for highly sensitive secrets.

Dynamic Options — source

Use source instead of options to populate a dropdown or checkbox from a live API. The request is made server-side, so tokens never reach the browser.

source:
  url: "https://{{env:DT_ENV_ID}}/api/v2/managementZones"
  headers:                          # optional
    Authorization: "Api-Token {{env:DT_API_KEY}}"
  path: "managementZones"           # dot-notation path to the array in the response
  value: "id"                       # field from each item used as the payload value
  label: "name"                     # field from each item shown in the UI
PropertyRequiredDescription
urlYesFull URL of the GET endpoint. Supports {{env:VAR}} placeholders.
headersNoHTTP headers. Use {{env:VAR}} to inject env variable values.
pathNoDot-notation path to the array in the response (e.g. managementZones or data.items). Omit if the response root is already an array.
valueNoField of each item to use as the option value in the payload. Omit to use the whole item.
labelNoField of each item to display in the UI. Omit to use the whole item.
